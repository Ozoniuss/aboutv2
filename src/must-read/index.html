Dump of things I found worth reading or watching.

Things I follow regularly:

  • The pragmatic engineer (Gergely Orosz) is the best publication for following the latest news in tech. It's pricey at 15$ a month but worth every penny IMO. Have a read before forwarding bullshit AI predictions from LinkedIn bros.

  • Modern Software Engineering teaches you how to build a great software development process as a company and how to improve as a software engineer. This is what I am personally looking for in a team.

  • Arjan Codes does a lot of videos focusing on good coding practices. They are mostly beginner-friendly but I still find them very enjoyable and easy to watch and he puts out content regularly.

  • Core Dumped makes the best videos I've seen so far explaining how programs and computers work at a low level.

  • Carlos Arguelles has a lot of articles, mostly focusing on comparing culture between Google and Amazon (and Microsoft to a small extent). But if you read their articles, do keep in mind that he's very biased towards Amazon. I still find them enjoyable.
    Please note that this is not a recommendation to get a Medium subscription, or read from Medium in general. Medium has some good authors but in general it's filled with garbage click-bait or rage-bait content. I only write on Medium because it was very simple to get started and publications help with reach, but I will give it up either for substack or my own blog if I get some time to set it up properly.

One time watch or reads:

  • Go's newsletter from 16th of January has a lot of good artices. Lots about concurrency models. Particularly, I'd have a go at:

    • Atomic vs mutex operations (even though the diagrams are kind of stupid). It's a good introduction to both and to operations optimized at the assembly level like compare-and-swap. Have a look at Go's atomic package for more details.

    • Mutexes from scratch is a great in-depth low-level article about how mutexes are implemented and why it's often a combination between runtime checks and OS-level system calls to use OS-level locks. It also introduced FUTEX as a concept as well as performance monitoring using the time Linux utility. This article was so good it made me follow the author.

    • Finding goroutine leaks was a great incident analysis and example of actions taken during the incident to mitigate it and after the incident to make the system more resilient. Tests and metrics are truly your friend. The part where I would have liked the author to Go into more details was how they actually deployed the fixes as they were talking about not being able to restart production they did make code changes and deploy them in order to mitigate, which makes me wonder how this could have been done with no restarts.

  • What color is your function? is an (older) rant from the Dart author about async/await. To be honest, I feel it's a valid rant. Personally, I always found it harder to reason about asynchronous programming than to explicitly write the logic in the style of traditional "threads" (I'm not talking about OS-level threads as most languages have some lightweight version of "threads" with cheap context-switch). Go is my favourite concurrency model, which seems to resonate with the author.

  • Most of the stuff from here for programming language design theory.

  • About name resolution. There's a lot that goes into figuring out which variable to actually use if you define multiple ones with the same name. And different languages handle it differently.

  • Gin is a very bad software library is an excellent read if you do web programming with Golang. The writing style of the author is very much to my linking, check out their other stuff.

  • Zig's migration to Codeberg is just another anecdote which shows the decline of Github.

  • Github's attempt of trying to charge for self-hosted runners. What a shit company.

  • Primeagen at Lex Fridman is a great success story from an average developer going from great struggles to success.

  • Benjamin Erb's thesis has some great stuff about the theory behind different concurrency models.

  • Concurrency is not parallelism by Rob Pike helps you build a great mental model of what concurrency is in programming languages. You will find it much easier to understand how you can have multiple threads running on a single core and also how you can have a single-threaded runtime (Javascript) use multiple cores.

  • Rob Pike's implementation of common generic functions like map, reduce and filter. And how he feels about them.
